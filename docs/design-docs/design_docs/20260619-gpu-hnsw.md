- **Created:** 2026-06-19
- **Author(s):** @6si
- **Status:** Implemented (GPU search + native fp32/fp16/bf16/int8 storage +
  CPU-parity filtered search; validated across the full dtype × metric matrix and
  delete-ratio range on GPU-indexed segments)
- **Component:** Index | QueryNode
- **Related Issues:** milvus-io/milvus#50653, zilliztech/knowhere#1686
- **Released:** N/A

## Feature Overview

### What

`GPU_HNSW` is a new index type that moves HNSW graph search from CPU to GPU.
Collections built with the standard CPU `HNSW` index can be searched on GPU at
load time via a configuration swap — no rebuild required. The GPU kernel searches
the same graph structure with identical recall characteristics, but leverages GPU
memory bandwidth (10-20x higher than CPU DRAM) to achieve large throughput
improvements.

Storage precision is preserved end-to-end: **FP32 / FP16 / BF16 / INT8** vectors
are uploaded to VRAM in their native layout (4 / 2 / 2 / 1 bytes per element), and
all three metrics (**L2 / IP / COSINE**) are supported. Filtered search (deletes /
TTL / partition bitset) runs on the GPU at CPU-HNSW parity.

### Why

Milvus users running large-scale HNSW workloads (100M-1B+ vectors) are
bottlenecked by CPU memory bandwidth during graph traversal. Existing GPU index
options have limitations:

- **CAGRA**: Self-match recall bug at N >= 500K due to insufficient graph degree
  ([rapidsai/cuvs#2102](https://github.com/rapidsai/cuvs/issues/2102)). Fixing
  requires 4x VRAM and 30% slower search. On our INT8/COSINE workload it returned
  0% recall.
- **GPU_IVF_PQ**: Crashes with `cudaErrorInvalidAddressSpace` on INT8 query
  vectors.
- **CPU HNSW**: Limited to ~1,600 vec/s per node at 500M+ scale due to DRAM
  bandwidth.

GPU_HNSW fills this gap: proven HNSW recall + GPU throughput, with zero index
rebuild cost.

### User Experience

1. Build a collection with standard `HNSW` index (no change to existing workflow).
2. Configure Milvus to load HNSW segments onto GPU:
   ```yaml
   knowhere:
     HNSW:
       load:
         override_index_type: GPU_HNSW
   ```
3. Load the collection — segments are automatically converted to GPU format.
4. Search as normal — queries are transparently routed to the GPU kernel.

No changes to SDK, query syntax, or search parameters (`ef` works identically).

### Performance

Benchmarked on production-scale INT8/COSINE HNSW (dim 384, ~500M+ rows). GPU HNSW
matches CPU-HNSW recall (R@10 ≈ 0.95) while the throughput ceiling moves from
CPU DRAM bandwidth (~1,600 vec/s/node) to the query client/network fanout; GPU
compute stays well under saturation, i.e. the kernel has headroom beyond the
measured throughput.

## Design

### Architecture

```
Milvus (Go orchestration) → Knowhere (C++/CUDA) → faiss GpuIndexHNSW → GPU hardware
```

**faiss** (`faiss/gpu/*`, re-vendored into knowhere `thirdparty/faiss`) owns the
`GpuIndexHNSW` search backend: a CPU-built `faiss::IndexHNSW` is uploaded via the
standard `index_cpu_to_gpu` cloner and searched with a parallel beam-search kernel
(recall tuned via `ef`). See facebookresearch/faiss#5459.

**Knowhere** owns:
- The `GPU_HNSW` / `GPU_HNSW_SQ` index nodes that wrap `GpuIndexHNSW`.
- FAISS HNSW graph → GPU flat-array conversion at load time.
- FP32 / FP16 / BF16 / INT8 upload in native precision; the kernel up-converts
  each element to fp32 for distance accumulation (int8 L2/IP uses DP4A).
- Warp-cooperative distance computation for bandwidth efficiency.
- Host- vs device-memory cost estimation (see Resource estimation below).

**Milvus** owns:
- Load-time index type swap (`HNSW` → `GPU_HNSW` via `override_index_type`),
  applied on all hot-load code paths (`LoadSegments` handler, `getLoadInfo()`,
  `ReopenSegments`, QueryCoord task load, QueryNode services).
- `GPU_HNSW` / `GPU_HNSW_SQ` param checker for collection-create validation.
- GPU VRAM admission (`gpu_memory_cost` threaded from knowhere through the C ABI).
- `StaticHasRawData=false` to skip redundant CPU embedding binlog loading.

### Index Lifecycle

```
┌─────────────┐     ┌──────────────────┐     ┌───────────────┐     ┌─────────────┐
│ Build (CPU) │────▶│ Store (S3/MinIO) │────▶│ Load (→GPU)   │────▶│ Search(GPU) │
│ standard    │     │ standard HNSW    │     │ convert+upload│     │ beam search │
│ HNSW build  │     │ binary format    │     │ to VRAM       │     │ kernel      │
└─────────────┘     └──────────────────┘     └───────────────┘     └─────────────┘
```

1. **Build (CPU):** Standard FAISS HNSW build — no pipeline change.
2. **Store:** Same binary format as CPU HNSW.
3. **Load (CPU → GPU):** if `override_index_type=GPU_HNSW`, knowhere deserializes
   the FAISS HNSW index on CPU, uploads the graph + vectors (native precision) to
   VRAM via `GpuIndexHNSW`, and the CPU copy is freed. Eager upload, no cold-start.
4. **Search (GPU):** Queries dispatched to the GPU kernel; results returned to CPU.
5. **Unload:** GPU memory freed on segment release.

### Storage & precision (native low precision — no decode-to-fp32)

| Storage (faiss) | Device layout | B/elem | Distance kernel |
|---|---|---|---|
| `IndexHNSWFlat` (fp32) | native fp32 | 4 | fp32 accumulation |
| `HNSW_SQ` `QT_8bit_direct_signed` | native int8 | 1 | DP4A (int8×int8→int32) |
| `HNSW_SQ` `QT_fp16` | native fp16 | 2 | `__half2float` → fp32 acc |
| `HNSW_SQ` `QT_bf16` | native bf16 | 2 | `__bfloat162float` → fp32 acc |

Vectors stay in their native layout on the device (up-converted to fp32 only
inside the distance computation), preserving the VRAM advantage at 500M+ scale.
FP16/BF16 mirror the INT8 precedent: 2 bytes/element on the device rather than
decoding to fp32 on upload (which would cost the same VRAM as fp32 and defeat the
reason to pick the low-precision type). Knowhere advertises `feature::FP16|GPU`
and `feature::BF16|GPU`, so the Milvus checker auto-accepts those dtypes through
the cgo feature flags (no Go logic change).

**INT8 cosine:** `QT_8bit_direct_signed` is a fixed `code = x + 128` map, so
L2-normalized (cosine) vectors collapse onto a few of 256 levels and lose recall.
INT8 **cosine** is therefore re-encoded to **fp16** at load; INT8 **L2/IP** keep
`direct_signed` + DP4A. Cosine inverse-L2-norms are computed from the decoded
values and applied at search time (stored codes are not normalized).

**Verification:** FP16/BF16 and int8-cosine storage/recall are GPU-runtime
validated by the faiss `TestGpuIndexHNSW` suite (`SQ_Fp16_Cosine`,
`SQ_Bf16_Cosine`, int8 gates) and the knowhere GPU HNSW UT suite (dtype × metric
recall vs a brute-force oracle, CUDA upload fault-injection). Measured recall:
fp16/bf16 track fp32 (≥0.974); int8 ≈0.92 (expected quantization degradation).

### Index-type registration (`GPU_HNSW` / `GPU_HNSW_SQ`)

`GPU_HNSW_SQ` is registered in knowhere as a thin subclass of the GPU HNSW index
node that only overrides `Type()`; the GPU search path is identical (both upload a
CPU-built HNSW / HNSW_SQ index to `GpuIndexHNSW`), so no separate kernel or storage
path is needed. Both types are registered for `VECTOR_FLOAT`, `VECTOR_FLOAT16`,
`VECTOR_BFLOAT16`, `VECTOR_INT8`.

A dedicated Go param checker (`gpuHnswChecker`) validates M / efConstruction /
metric / data-type ranges and is routed ahead of the unified vector checker for
`GPU_HNSW` / `GPU_HNSW_SQ` (the unified checker's knowhere `ValidateIndexParams`
is a no-op for these types, so without dedicated routing the range validation
would never run).

### Configuration

```yaml
# milvus.yaml — enable GPU HNSW
knowhere:

    load:
      override_index_type: GPU_HNSW

# Optional: segment sizing for GPU occupancy

  segment:
    maxSize: 4096  # targets ~8M rows/segment for good GPU occupancy
```

### Resource estimation & VRAM admission

GPU segments must be admitted against **device** memory, not host RAM. Knowhere
reports host and device cost separately (`Resource{ maxMemoryCost /* host staging
peak */, gpuMemoryCost /* device-resident */ }`); Milvus threads `gpuMemoryCost`
through `resource_c.h` → `IndexFactory.cpp` → `segment_loader.go`, which admits
GPU loads against VRAM (falling back to the host estimate when the GPU cost is 0).

The GPU reservation is conservative (≈2× file size) so admission never
under-reserves; measured int8-cosine **resident** usage is ≈1.7× file size (fp16
codes ≈2×; the HNSW graph is unchanged), so a node stops/​rebalances before it can
overcommit VRAM — no OOM under balanced placement.

### Compaction Compatibility

GPU_HNSW segments compact through the standard datacoord path; no compaction,
routing, or delete-pipeline code assumes GPU segments are delete-free.

## Filtered search (delete / TTL / partition bitset) — CPU parity

Milvus collapses delete / TTL / partition state into a per-segment `BitsetView`
(bit set ⇒ row filtered out) passed into `Search()`. GPU_HNSW consumes the **same**
bitset and reproduces the CPU-HNSW filtered-search semantics on the GPU; no
Milvus-side delete / L0 / TSafe logic changes.

**Three mechanisms (all required for parity):**

1. **Two-tier traversal.** The kernel keeps a valid result beam (non-filtered
   nodes → top-k) and a separate invalid frontier (filtered nodes retained
   **only** for expansion, while their distance beats the valid beam's worst).
   Filtered nodes are still traversed as graph waypoints so recall is preserved,
   but never enter the result set. Validity is packed into the existing
   `is_expanded` word (no new per-slot array); the result beam is byte-for-byte
   today's beam at 0% filter, so recall cannot regress.
2. **Alpha admission gate.** Mirrors CPU's `accumulated_alpha` (`+= kAlpha; if
   (<1) skip; else -= 1`, `kAlpha = filter_ratio * 0.7`), run single-threaded in
   the already-serial post-sort section over the distance-sorted candidates.
   Deterministic and preserves the rate-limiting effect in distance order; parity
   is asserted empirically by the recall gate, not by identical traversal order.
3. **Brute-force fallback.** Up-front (host): if `filtered ≥ 0.93·N` or
   `k ≥ 0.5·live`, skip the graph and scan live rows directly. Per-query
   (device): the graph kernel atomically appends queries whose `valid_count < k`
   (with `≥ k` live rows) to a `needs_bf[]` list, then one batched BF kernel runs
   over exactly those queries on the same stream — guaranteeing `k` valid results
   without a mid-search host round-trip. Honors `disable_fallback_brute_force`.

**ID-space invariant (load-bearing):** for a single `IndexHNSWFlat` / `HNSW_SQ`
segment, storage id == add order == segment row offset == `BitsetView` index, so
`bitset.test(node_id)` is a direct 1:1 lookup (asserted by a dedicated test
before relying on it).

**Non-goals:** partition-key multi-index routing; iterator / range-search filtered
paths on GPU (knn only in phase 1).

**Bitset upload:** identical for all `nq` queries → uploaded once per `Search()`
into a per-scratch-pool-slot `d_bitset` (read-only in the kernel, so concurrent
filtered searches stay isolated). VRAM cost is negligible (~125 KB/1M rows).

### Failure Modes

| Scenario | Behavior |
|---|---|
| VRAM exhaustion | Load admission reserves conservatively (≈2× file); a node rebalances before overcommit. In mixed mode queries fall back to CPU nodes. |
| GPU hardware failure | Segment marked unhealthy; querycoord redistributes to other nodes. |
| Unsupported metric | Param checker rejects at create time (only L2, IP, COSINE supported). |
| Filtered search (delete / TTL / partition) | **Supported (CPU parity)** — two-tier beam + alpha gate + BF fallback (above). Filtered rows absent from results, live rows present, recall matches CPU. |
| Vector output / raw-data reconstruction | **Safe — served from raw field data, not the index.** `GetVectorByIds` returns `not_implemented` and `HasRawData()` is `false`; because `IndexHasRawData(field_id)` is false, segcore eager-loads raw field data and routes output-vector retrieval through `get_raw_data`, never the index. Index-side raw-vector refine (iterator / range-search) is not offered for GPU_HNSW. |

## Design history

- The initial kernel shipped with an "Overflow Candidate Queue" (OCQ) that was
  never wired up (dead code); it was removed in favor of a plain parallel
  beam-search kernel with recall tuned via `ef`. No behavioral change (the queue
  was always empty).
- Filtered search was initially rejected (`invalid_args`) and GPU_HNSW scoped to
  append-mostly / immutable collections; GPU-side bitset filtering (above) was
  since implemented and validated at CPU parity, lifting that limitation.
