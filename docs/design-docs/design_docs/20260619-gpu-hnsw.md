- **Author(s):** @6si

## Feature Overview

### What

`GPU_HNSW` is a new index type that moves HNSW graph search from CPU to GPU. Collections built with the standard CPU `HNSW` index can be searched on GPU at load time via a configuration swap — no rebuild required. The GPU kernel searches the same graph structure with identical recall characteristics, but leverages GPU memory bandwidth (10-20x higher than CPU DRAM) to achieve 10-50x throughput improvement.

### Why

Milvus users running large-scale HNSW workloads (100M-1B+ vectors) are bottlenecked by CPU memory bandwidth during graph traversal. Existing GPU index options have limitations:

- **CAGRA**: Self-match recall bug at N >= 500K due to insufficient graph degree ([rapidsai/cuvs#2102](https://github.com/rapidsai/cuvs/issues/2102)). Fixing requires 4x VRAM and 30% slower search.
- **GPU_IVF_PQ**: Crashes with `cudaErrorInvalidAddressSpace` on INT8 query vectors.
- **CPU HNSW**: Limited to ~2-4K QPS at 500M+ scale due to DRAM bandwidth.

GPU_HNSW fills this gap: proven HNSW recall + GPU throughput, with zero index rebuild cost.

> **✅ Filtered search — supported (CPU parity).** A GPU_HNSW / GPU_HNSW_SQ segment
> applies the `BitsetView` that Milvus uses for deletes, TTL expiry, and
> partition/visibility filters. Filtered rows are excluded from results but are
> still traversed as graph waypoints (so recall is preserved), and the kernel
> falls back to a brute-force scan over live rows at high filter ratios — matching
> the CPU HNSW semantics in Knowhere. Earlier revisions rejected a non-empty
> bitset with `invalid_args` and scoped GPU_HNSW to append-mostly / immutable
> collections; **that limitation is lifted.** See the design doc
> [20260718-gpu-hnsw-filtered-search.md](20260718-gpu-hnsw-filtered-search.md),
> the [Failure Modes](#failure-modes) table, and the
> [OCQ-removal / limitations decision record](20260711-gpu-hnsw-ocq-removal.md)
> (the filtered-search decision there is now superseded).

### User Experience

1. Build a collection with standard `HNSW` index (no change to existing workflow)
2. Configure Milvus to load HNSW segments onto GPU:
   ```yaml
   knowhere:
     HNSW:
       load:
         override_index_type: GPU_HNSW
   ```
3. Load the collection — segments are automatically converted to GPU format
4. Search as normal — queries are transparently routed to the GPU kernel

No changes to SDK, query syntax, or search parameters (`ef` works identically).

### Performance

Benchmarked on 529M x 1024-dim INT8 vectors (production workload):

| Configuration | Throughput | R@10 | Latency (p50, batch=1000) |
|---|---|---|---|
| CPU HNSW, 8 nodes | ~1,100 vec/s | 0.950 | 920ms |
| CPU HNSW, 16 nodes | ~4,000 vec/s | 0.950 | 330ms |
| GPU HNSW, 8 GPU nodes (projected) | ~35,000 vec/s | 0.950 | ~30ms |

## Design

### Architecture

Two layers handle GPU HNSW end-to-end:

```
Milvus (Go orchestration) → Knowhere (C++/CUDA kernel) → GPU hardware
```

**Knowhere** owns:
- CUDA parallel beam-search kernel (recall tuned via `ef`)
- FAISS HNSW graph → GPU flat-array conversion at load time
- FP32 / FP16 / BF16 / INT8 data upload to VRAM in the vectors' native
  precision (4 / 2 / 2 / 1 bytes per element); the search kernel up-converts
  each element to fp32 for distance accumulation
- Warp-cooperative distance computation for bandwidth efficiency

**Milvus** owns:
- Load-time index type swap (`HNSW` → `GPU_HNSW` via `override_index_type`)
- Applied on all 5 hot-load code paths:
  1. `LoadSegments` handler entry point
  2. `executor.getLoadInfo()` 
  3. `ReopenSegments` path
  4. QueryCoord task load path
  5. QueryNode services handler
- `GPU_HNSW` param checker for collection creation validation
- `StaticHasRawData` to skip redundant CPU embedding binlog loading

### Index Lifecycle

```
┌─────────────┐     ┌──────────────────┐     ┌──────────────┐     ┌─────────────┐
│ Build (CPU) │────▶│ Store (S3/MinIO) │────▶│ Load (→GPU)  │────▶│ Search(GPU) │
│ standard    │     │ standard HNSW    │     │ convert+upload│     │ beam search │
│ HNSW build  │     │ binary format    │     │ to VRAM      │     │ kernel      │
└─────────────┘     └──────────────────┘     └──────────────┘     └─────────────┘
```

1. **Build (CPU):** Standard FAISS HNSW build — no changes to the existing pipeline
2. **Store:** Index stored in the same binary format as CPU HNSW
3. **Load (CPU → GPU):** On segment load, if `override_index_type=GPU_HNSW`:
   - Knowhere deserializes the FAISS HNSW index on CPU
   - Converts adjacency lists to GPU-friendly flat arrays (CSR-like format)
   - Uploads the vectors (FP32 flat, or FP16 / BF16 / INT8 from HNSW_SQ storage,
     each kept in its native low-precision layout) and graph structure to GPU VRAM
   - Segment becomes GPU-searchable immediately (eager upload, no cold-start)
4. **Search (GPU):** Queries dispatched to GPU kernel, results returned to CPU
5. **Unload:** GPU memory freed on segment release

### Configuration

```yaml
# milvus.yaml — enable GPU HNSW
knowhere:
  HNSW:
    load:
      override_index_type: GPU_HNSW

# Optional: segment sizing for optimal GPU throughput
dataCoord:
  segment:
    maxSize: 4096  # targets ~8M rows/segment for good GPU occupancy
```

### Compaction Compatibility

GPU_HNSW segments compact through the standard datacoord path. Fallback compaction
start-position handling (`normalizePositionTimestamp` in `internal/datacoord/meta.go`,
which bumps a bulk-imported segment's fallback position upward rather than rejecting)
is upstream behavior and is **not** modified by this feature — GPU_HNSW segments
simply inherit it. (An earlier draft of this doc claimed this feature introduced that
change; it does not, and `meta.go` is not in this PR.)

### Failure Modes

| Scenario | Behavior |
|---|---|
| VRAM exhaustion | Segment load fails with OOM; query falls back to CPU nodes in mixed mode |
| GPU hardware failure | Segment marked unhealthy; querycoord redistributes to other nodes |
| Unsupported metric | Param checker rejects at create time (only L2, IP, COSINE supported) |
| Filtered search (delete / TTL / partition bitset) | **Supported (CPU parity).** A GPU_HNSW/GPU_HNSW_SQ segment applies the `BitsetView`: filtered rows are excluded from results but still traversed as graph waypoints so recall is preserved, and the kernel falls back to a brute-force scan over live rows at high filter ratios. Validated at CPU-HNSW parity across L2/IP/COSINE and delete ratios on sealed, GPU-indexed segments (deleted rows absent, live rows present, no `invalid_args`). See [20260718-gpu-hnsw-filtered-search.md](20260718-gpu-hnsw-filtered-search.md). |
| Vector output / raw-data reconstruction | **Safe — served from raw field data, not the index.** `GpuHnswIndexNode::GetVectorByIds` returns `not_implemented` and `HasRawData()` is `false` once the vectors are uploaded to GPU (the CPU copy is freed). This does not break vector-field output: because `IndexHasRawData(field_id)` is `false`, segcore (a) eager-loads the raw field data at load time (`SegmentLoadInfo`: `!index_has_raw_data ⇒ should_eager_load`) and (b) routes `bulk_subscript`/output-vector retrieval through `get_raw_data` (column data), never through the index's `GetVectorByIds` (`ChunkedSegmentSealedImpl.cpp` — the `IndexHasRawData` branch guards the `get_vector` call). Iterator/range-search refine that relies on index-side raw-vector reconstruction is not offered for GPU_HNSW. |

## Dependencies

- **Knowhere PR:** [zilliztech/knowhere#1686](https://github.com/zilliztech/knowhere/pull/1686) — GPU HNSW beam-search kernel with native FP32 / FP16 / BF16 / INT8 support
- **Hardware:** NVIDIA GPU with compute capability 7.0+ (Volta or newer), CUDA 12.x+
- **Runtime:** `libatomic1` added to GPU Dockerfile
